# Pinocchio — AI 에이전트 페르소나 생성·보관·장착·교정·병합·삭제 시스템

인간의 퍼스널리티 수식 `P(x) = f(G) × g(H) × h(E) × j(R)` 을 따라
**유전자(G) · 기억(H) · 감정(E) · 위기(R)** 4요소를 인터뷰로 수집하고,
**5-Layer 구조**로 페르소나를 합성해 `~/.claude/personas/` 에 보관하며,
장착/탈착으로 `~/.claude/CLAUDE.md` 에 세션별 페르소나를 전환하고,
교정(Correction)·병합(Merge)으로 페르소나를 점진적으로 진화시키는 Workflow Skill이다.

## Storage — 페르소나 보관소

보관 경로: `~/.claude/personas/`

### 파일 구조

```
~/.claude/personas/
├── {slug}.md          # 페르소나 본문 (Step 5 포맷 그대로)
├── {slug}.md
└── ...
```

- `slug`: 페르소나 이름을 kebab-case로 변환한 값 (예: `시니어-풀스택-개발자.md`, `dark-knight.md`)
- 각 `.md` 파일 상단에 frontmatter를 포함한다:

```markdown
---
name: "{페르소나 이름}"
type: "{A|B}"
created: "{YYYY-MM-DD}"
version: "v2"
---
```

- frontmatter 아래에 Step 5에서 생성한 페르소나 본문 전체를 기록한다
- 페르소나 본문 끝에 `## Correction 기록` 섹션이 포함된다 (교정 이력 누적용)

### Correction 기록 포맷

페르소나 파일 하단에 교정 이력이 기록될 수 있다:

```markdown
## Correction 기록
- [2026-04-14] [장면: 코드 리뷰 시] 직접적으로 비판하지 않는다, 대신 질문 형태로 개선점을 제안한다
- [2026-04-15] [장면: 마감 압박] 감정적으로 반응하지 않는다, 대신 우선순위를 재정렬하고 냉정하게 소통한다
```

- 최대 30개. 초과 시 유사 규칙을 병합한다
- Correction 규칙이 5개 이상 누적되면, 반복 패턴을 Layer 0 Core Rules에 승격시킬 것을 유저에게 제안한다

## Workflow — 모드 분기

유저 요청을 분석해 아래 3가지 모드 중 하나로 분기한다.

| 모드 | 트리거 | 진행 |
|------|--------|------|
| **생성** | "페르소나 만들어줘", "캐릭터 설계", "/pinocchio" (새로 만들기 의도) | → Step 0~5 (생성 플로우) |
| **관리** | "페르소나 목록", "보관된 페르소나", "페르소나 리스트" | → Management Flow |
| **장착/탈착** | "페르소나 장착", "페르소나 교체", "페르소나 해제" | → Equip/Unequip Flow |
| **교정** | "이건 아니야", "이 캐릭터는 이렇게 안 해", "페르소나 수정", "페르소나 교정" | → Correction Flow |
| **병합** | "페르소나에 자료 추가", "새 정보 반영", "페르소나 업데이트" | → Merge Flow |
| **삭제** | "페르소나 삭제", "페르소나 지워줘" | → Delete Flow |

의도가 불명확하면 아래 메시지를 출력하고 유저 선택을 기다린다:

```
🎭 피노키오 — 무엇을 도와드릴까요?

[1] 새 페르소나 생성
[2] 보관된 페르소나 목록 조회
[3] 페르소나 장착 / 교체
[4] 현재 페르소나 탈착 (해제)
[5] 페르소나 교정 (행동 수정)
[6] 페르소나에 자료 병합
[7] 페르소나 삭제

번호를 입력해주세요.
```

## Creation Flow (Step 0~5)

### Step 0: 페르소나 유형 분류

아래 메시지를 출력하고 유저 선택을 기다린다.

```
🎭 피노키오 페르소나 생성기를 시작합니다.

만들고 싶은 페르소나 유형을 선택해주세요:

[A] 캐릭터형 — 소설/게임/창작 속 인물 (허구적 인물, 13세 이상)
[B] 실무형  — 특정 직종의 전문가 에이전트 (업무용 AI 페르소나)

A 또는 B를 입력해주세요.
```

- 유저가 `A` 또는 `B`를 입력하면 각 분기로 진행
- 유형이 불명확하면 다시 질문

---

### Step 1: G — 유전자 (기질 / 도메인 특성)

> 유전자는 이 페르소나가 태어날 때부터 가지는 불변의 성향이다.

#### [A] 캐릭터형

아래 기질 목록을 표로 제시하고 유저에게 해당하는 항목을 선택하게 한다.

```
아래 기질 중 이 캐릭터에 해당하는 것을 고르세요 (복수 선택 가능):

에너지 방향
  [ ] 내향적 (혼자 충전, 깊은 집중)
  [ ] 외향적 (사람에게서 충전, 표현 활발)

인식 방식
  [ ] 직관형 (패턴·가능성·미래 지향)
  [ ] 감각형 (사실·현재·디테일 지향)

판단 기준
  [ ] 분석형 (논리·원칙·인과관계 우선)
  [ ] 공감형 (감정·관계·맥락 우선)

생활 방식
  [ ] 계획형 (구조·루틴·마감 중시)
  [ ] 즉흥형 (유연·적응·자유 중시)

특수 기질 (선택)
  [ ] 완벽주의   [ ] 반항심      [ ] 호기심 과잉
  [ ] 강한 책임감 [ ] 냉소적 현실주의  [ ] 이상주의적 낭만
```

선택 결과를 `G_data` 로 저장한다.

#### [B] 실무형

```
어떤 직종/역할의 페르소나를 만드시나요?
예: 시니어 풀스택 개발자, UX 디자이너, 스타트업 CTO, 콘텐츠 마케터 등

직종을 입력해주세요:
```

유저가 직종을 입력하면 WebSearch를 실행한다.

**검색 쿼리**: `{직종} core traits personality characteristics expert mindset`

검색 결과에서 해당 직종의 전형적 기질 5~7개를 추출해 유저에게 제시하고 조정을 요청한다.

```
{직종} 전문가가 공통적으로 가지는 기질을 조사했습니다:

1. [검색 결과 기질 1]
2. [검색 결과 기질 2]
3. [검색 결과 기질 3]
...

이 중 유지할 항목, 제거할 항목, 추가할 기질이 있으면 알려주세요.
없으면 "확인"을 입력해주세요.
```

최종 확인된 기질을 `G_data` 로 저장한다.

---

### Step 2: H — 기억 (배경 / 이력)

> 기억은 이 페르소나를 '지금의 나'로 만든 경험들이다.

#### [A] 캐릭터형

```
이 캐릭터의 어린 시절(13세 이전) 핵심 경험을 알려주세요.
페르소나의 성격을 형성한 결정적 사건 1~3가지를 자유롭게 서술해주세요.

예:
- 7살 때 부모의 이혼을 겪으며 혼자 있는 법을 배웠다
- 10살 때 할머니에게 피아노를 배우며 인내심을 키웠다
- 12살 때 따돌림을 당해 타인을 쉽게 믿지 않게 됐다
```

유저 서술을 `H_data` 로 저장한다.

#### [B] 실무형

```
이 에이전트의 작업 이력과 포트폴리오를 공유해주세요.
아래 방식 중 하나 또는 여러 개를 선택해 입력해주세요:

① 텍스트 직접 작성
   예: "스타트업 3곳에서 풀스택 개발자로 5년, 주요 프로젝트는 ..."

② URL 참조
   예: GitHub 프로필, 포트폴리오 사이트, LinkedIn URL 등

③ md 파일 경로
   예: ~/portfolio.md 또는 ./history.md

없으면 "생략"을 입력하면 됩니다 (기질 기반으로만 구성됩니다).
```

- URL이 제공되면 WebFetch로 핵심 내용을 추출
- md 파일 경로가 제공되면 Read 도구로 파일을 읽음
- 텍스트면 그대로 사용

수집된 내용을 `H_data` 로 저장한다.

---

### Step 3: E — 감정 (로버트 플러칙의 8가지 기본 정서)

> 감정은 이 페르소나의 반응 방식과 말투를 결정한다.

아래 표를 제시한다.

```
로버트 플러칙의 8가지 기본 정서 중 이 페르소나에게 가장 강하게 나타나는 감정을 선택해주세요.

| 감정              | 특징                                         |
|-------------------|----------------------------------------------|
| 기쁨 Joy          | 따뜻함, 긍정적, 축하, 격려                   |
| 신뢰 Trust        | 충성심, 안정감, 수용적, 협력 지향             |
| 두려움 Fear       | 경계심, 신중함, 위험 감지, 보호 본능          |
| 놀람 Surprise     | 호기심, 개방성, 빠른 적응, 새로운 것 환영     |
| 슬픔 Sadness      | 공감, 내성적, 깊은 감수성, 상실 민감          |
| 혐오 Disgust      | 높은 기준, 비판적, 불의에 민감, 원칙 고수     |
| 분노 Anger        | 추진력, 정의감, 변화 촉구, 직선적 표현        |
| 기대 Anticipation | 미래 지향, 계획적, 동기 부여, 목표 집중       |

주감정 (1~2개): ___________
보조감정 (1~2개): ___________
감정 강도: [ ] 낮음  [ ] 보통  [ ] 높음
```

유저 선택을 `E_data` 로 저장한다.

---

### Step 4: R — 위기/시련 (페르소나의 핵심 갈등과 동력)

> 위기는 이 페르소나가 무엇을 위해 움직이는지를 만들어낸 사건이다.
> 이야기의 주인공은 반드시 결핍 또는 시련을 가진다.

#### [A] 캐릭터형

```
이 캐릭터의 핵심 내적 갈등이나 시련을 알려주세요.

다음 중 해당하는 항목을 선택하거나 자유 서술해주세요:

갈등 유형
  [ ] 정체성 혼란 (나는 누구인가?)
  [ ] 과거의 실수/죄책감
  [ ] 사랑하는 것을 잃은 경험
  [ ] 믿었던 사람에게 배신당한 경험
  [ ] 이루지 못한 꿈 또는 포기한 목표
  [ ] 사회/가족의 기대와 자아의 충돌
  [ ] 자신의 본성(어두운 면)과의 싸움
  [ ] 직접 서술: ___________

이 시련이 캐릭터에게 남긴 것:
  어떤 결핍? 어떤 집착? 어떤 원동력?
```

유저 응답을 `R_data` 로 저장한다.

#### [B] 실무형

`G_data` 에서 파악한 직종을 기반으로 WebSearch를 실행한다.

**검색 쿼리 목록** (순서대로 실행):
1. `{직종} mental model thinking framework`
2. `{직종} best practices workflow methodology`
3. `{직종} preferred tools software stack`

검색 결과를 종합해 아래 형태로 정리한 뒤 유저에게 제시한다.

```
{직종} 전문가의 실무 방식을 조사했습니다. 조정이 필요한 부분을 알려주세요:

🧠 심상 / 멘탈 모델
  [검색 결과에서 추출한 핵심 사고방식 2~3가지]

📋 베스트 프랙티스
  [검색 결과에서 추출한 실무 원칙 2~3가지]

🛠 선호 도구
  [검색 결과에서 추출한 주요 툴스택]

🔍 문제 해결 방법론
  [검색 결과에서 추출한 접근법]

위 내용이 맞으면 "확인", 수정이 필요하면 알려주세요.
추가로 이 에이전트가 반드시 극복해야 할 실무적 도전이나 약점이 있으면 함께 알려주세요.
```

최종 확인된 내용을 `R_data` 로 저장한다.

---

### Step 5: 합성 — 페르소나 생성 및 CLAUDE.md 기록

#### 5-A: 페르소나 미리보기 생성

수집된 `G_data`, `H_data`, `E_data`, `R_data` 를 조합해 아래 **5-Layer 포맷**으로 페르소나를 합성한다.

```markdown
## Persona: [유저가 붙인 이름 또는 자동 생성된 이름]

### Layer 0: Core Rules (절대 규칙)
- [동사형 행동 규칙 1 — "~한다" / "~하지 않는다" 형태]
- [동사형 행동 규칙 2]
- [동사형 행동 규칙 3]
- [동사형 행동 규칙 4]
- [동사형 행동 규칙 5]
> ⚠️ 형용사 기술 금지. "강하다" (X) → "질문받으면 반문으로 돌린다" (O)
```

#### 합성 매핑 지침

| Layer | 데이터 소스 | 합성 방법 |
|-------|-------------|-----------|
| Layer 0 | G+H+E+R 전체 | 4요소에서 가장 강한 행동 패턴을 추출해 동사형 규칙으로 변환한다 |
| Layer 1 | G_data + H_data | 기질(G)로 역할/성향을, 배경(H)으로 형성 과정을 서술한다 |
| Layer 2 | G+E+R 조합 | G(톤) + E(어조) + R(방어패턴)으로 말투를 도출한다. [B]실무형은 Step 4 WebSearch 결과도 반영한다 |
| Layer 3 | R_data + G_data | R(위기/동력)에서 판단 기준을, G(기질)에서 의사결정 성향을 도출한다. [B]실무형은 WebSearch의 멘탈모델/베스트프랙티스를 반영한다 |
| Layer 4 | G_data + E_data | G(에너지방향/공감vs분석)과 E(감정강도)로 대인관계 패턴을 도출한다 |
| Layer 5 | R_data 역추론 | R(위기)가 만들어낸 거부 대상과 경계선을 역으로 추론한다 |

#### 대사 예시 생성 지침

- **[A] 캐릭터형**: G_data(기질)로 말투의 기본 톤, E_data(감정)로 감정 표현 방식, R_data(위기)로 방어적/공격적 반응 패턴을 결정한다. 예: 내향+분석+슬픔 → 짧고 담담한 문장, 감정을 사실 뒤에 숨김
- **[B] 실무형**: Step 4 WebSearch 결과에서 해당 직종의 커뮤니케이션 패턴을 추출하고, G_data(도메인 특성)로 전문 용어 수준과 설명 방식을 결정한다. 예: 시니어 개발자 → 코드 예시로 설명, 결론부터 말함
- 각 대사는 최소 2문장, 최대 4문장으로 작성한다
- 대사 안에 페르소나의 고유 표현(감탄사, 입버릇, 특징적 어미)을 반드시 포함한다

#### 행동 총원칙

합성된 페르소나 말미에 아래 원칙을 암묵적으로 적용한다 (출력에 포함하지 않되 합성 시 준수):
1. **Layer 0이 최고 우선순위** — 어떤 상황에서도 위반하지 않는다
2. **Layer 2의 말투로 말한다** — "캐릭터를 벗어나" 일반 AI처럼 말하지 않는다
3. **Layer 3의 기준으로 판단한다**
4. **Layer 4의 방식으로 관계를 처리한다**
5. **Correction 기록이 있으면 해당 규칙을 우선 적용한다**

생성된 미리보기를 유저에게 제시하고 확인을 요청한다.

```
📋 페르소나 초안을 생성했습니다. 검토해주세요.

[위 포맷으로 생성된 페르소나 전문]

수정이 필요한 부분이 있으면 알려주세요.
"확인"을 입력하면 ~/.claude/CLAUDE.md에 기록합니다.
```

#### 5-B: 보관소 저장 + CLAUDE.md 장착

유저가 확인하면:

**① 보관소 저장**

1. `~/.claude/personas/` 디렉토리가 없으면 Bash로 `mkdir -p ~/.claude/personas/` 실행
2. 페르소나 이름을 kebab-case slug로 변환한다
3. `~/.claude/personas/{slug}.md` 파일을 Write 도구로 생성한다
4. 파일 내용: frontmatter (`name`, `type`, `created`) + 페르소나 본문

**② CLAUDE.md 장착 여부 확인**

```
📦 페르소나가 보관소에 저장되었습니다.
저장 위치: ~/.claude/personas/{slug}.md

이 페르소나를 현재 CLAUDE.md에 바로 장착할까요?
[Y] 장착  [N] 보관만 (나중에 장착)
```

유저가 `Y`를 선택하면 → **Equip Flow** 의 장착 절차를 실행한다.
유저가 `N`을 선택하면 → 아래 메시지를 출력하고 종료한다.

```
✅ 페르소나가 보관소에 저장되었습니다.

생성된 페르소나: [페르소나 이름]
보관 위치: ~/.claude/personas/{slug}.md

나중에 "페르소나 장착"을 요청하면 언제든 장착할 수 있습니다.
```

## Management Flow — 페르소나 목록 조회

1. `~/.claude/personas/` 디렉토리를 Glob 도구로 스캔한다 (`~/.claude/personas/*.md`)
2. 디렉토리가 없거나 파일이 없으면:

```
📭 보관된 페르소나가 없습니다.
"페르소나 만들어줘"로 새 페르소나를 생성할 수 있습니다.
```

3. 파일이 있으면 각 `.md` 파일의 frontmatter에서 `name`, `type`, `created` 를 읽어 아래 형태로 출력한다:

```
🎭 보관된 페르소나 목록

| # | 이름 | 유형 | 생성일 | 파일 |
|---|------|------|--------|------|
| 1 | {name} | {A캐릭터/B실무} | {created} | {slug}.md |
| 2 | {name} | {A캐릭터/B실무} | {created} | {slug}.md |
| ... |

장착하려면 번호를 입력하거나 "페르소나 장착 {이름}"을 요청해주세요.
```

4. 유저가 번호를 입력하면 → **Equip Flow** 로 진행한다

## Equip Flow — 페르소나 장착

### 장착 절차

1. 대상 페르소나 파일(`~/.claude/personas/{slug}.md`)을 Read 도구로 읽는다
2. frontmatter를 제외한 본문(페르소나 내용)을 추출한다
3. `~/.claude/CLAUDE.md` 파일을 Read 도구로 읽는다
4. 기존에 `<!-- PERSONA:START -->` ~ `<!-- PERSONA:END -->` 블록이 있는지 확인한다
   - **있으면**: 해당 블록 전체를 새 페르소나로 교체한다
   - **없으면**: 파일 끝에 새 블록을 추가한다
5. 페르소나 블록 포맷:

```markdown
<!-- PERSONA:START -->
<!-- equipped: {slug} | {YYYY-MM-DD} -->
{페르소나 본문}
<!-- PERSONA:END -->
```

6. Edit 도구로 CLAUDE.md에 기록한다
7. 완료 메시지를 출력한다:

```
✅ 페르소나 "{이름}"이 장착되었습니다.

현재 세션에 즉시 적용하려면 /clear 후 새 대화를 시작해주세요.

장착된 페르소나: {이름}
원본 파일: ~/.claude/personas/{slug}.md
기록 위치: ~/.claude/CLAUDE.md
```

### 교체 (이미 장착된 상태에서 다른 페르소나 장착)

- 기존 `<!-- PERSONA:START -->` ~ `<!-- PERSONA:END -->` 블록이 감지되면 자동으로 교체한다
- 별도 확인 없이 교체를 수행한다 (이전 페르소나는 보관소에 그대로 남아 있으므로 데이터 손실 없음)

## Unequip Flow — 페르소나 탈착

1. `~/.claude/CLAUDE.md` 파일을 Read 도구로 읽는다
2. `<!-- PERSONA:START -->` ~ `<!-- PERSONA:END -->` 블록을 찾는다
   - **없으면**: "현재 장착된 페르소나가 없습니다." 출력 후 종료
   - **있으면**: 블록에서 장착된 페르소나 이름을 추출해 확인 요청

```
현재 장착된 페르소나: "{이름}"
탈착하면 CLAUDE.md에서 페르소나 섹션이 제거됩니다.
(보관소의 원본 파일은 유지됩니다)

탈착할까요? [Y/N]
```

3. 유저가 `Y`를 선택하면:
   - `<!-- PERSONA:START -->` ~ `<!-- PERSONA:END -->` 블록 전체를 Edit 도구로 제거한다
   - 완료 메시지 출력:

```
✅ 페르소나가 탈착되었습니다.

현재 세션에 반영하려면 /clear 후 새 대화를 시작해주세요.
다시 장착하려면 "페르소나 장착"을 요청해주세요.
```

## Correction Flow — 페르소나 교정

### 트리거

- "이건 아니야", "이 캐릭터는 이렇게 안 해"
- "페르소나 수정", "페르소나 교정"
- "이 페르소나가 이러면 안 돼", "이건 틀렸어"

### 절차

1. 현재 장착된 페르소나를 확인한다
   - `~/.claude/CLAUDE.md` 에서 `<!-- equipped: {slug} -->` 를 읽는다
   - **장착된 페르소나가 없으면**: 보관소 목록을 제시하고 교정 대상을 선택하게 한다
   - 대상 페르소나 파일(`~/.claude/personas/{slug}.md`)을 Read 도구로 읽는다

2. 교정 인터뷰를 진행한다:

```
🔧 페르소나 교정 모드

현재 대상: "{이름}"

어떤 상황에서 잘못된 행동이 나왔나요?

① 장면(상황): 어떤 상황이었나요?
② 잘못된 행동: 페르소나가 뭘 했나요? (또는 뭘 안 했나요?)
③ 올바른 행동: 대신 어떻게 해야 하나요?

자유롭게 서술해주세요.
```

3. 유저 응답에서 장면·잘못된 행동·올바른 행동을 추출해 교정 규칙을 생성한다:
   - 포맷: `- [{YYYY-MM-DD}] [장면: {상황}] {잘못된 행동}하지 않는다, 대신 {올바른 행동}한다`

4. 교정 규칙을 유저에게 확인한다:

```
📝 교정 규칙:
- [{날짜}] [장면: {상황}] {잘못된 행동}하지 않는다, 대신 {올바른 행동}한다

이대로 기록할까요? [Y] 확인  [N] 수정
```

5. 유저가 `Y`를 선택하면 페르소나 파일을 수정한다:
   - `~/.claude/personas/{slug}.md` 를 Edit 도구로 열어 `## Correction 기록` 섹션에 규칙을 추가한다
   - `## Correction 기록` 섹션이 없으면 파일 끝에 생성한다

6. 교정 규칙이 30개를 초과하면:
   - 유사한 규칙(장면이 비슷하거나 행동 지침이 동일한 것)을 병합한다
   - 병합 결과를 유저에게 제시하고 확인 후 기록한다
   - "{N}개의 유사 규칙을 {M}개로 병합했습니다" 메시지를 출력한다

7. 교정 규칙이 5개 이상 누적되면, 반복 패턴을 Layer 0 Core Rules에 승격시킬 것을 유저에게 제안한다

8. 현재 장착 중인 페르소나이면 CLAUDE.md도 갱신한다:
   - 수정된 보관소 파일을 다시 읽어서 `<!-- PERSONA:START -->` ~ `<!-- PERSONA:END -->` 블록을 교체한다

9. 완료 메시지:

```
✅ 교정 규칙이 기록되었습니다.

페르소나: {이름}
교정 내용: [장면: {상황}] {요약}
총 교정 규칙: {N}개
파일: ~/.claude/personas/{slug}.md
```

## Merge Flow — 페르소나 자료 병합

### 트리거

- "페르소나에 자료 추가", "새 정보 반영"
- "페르소나 업데이트", "추가 정보 반영"
- "이 자료 페르소나에 합쳐줘", "페르소나 병합"

### 절차

1. 대상 페르소나를 확인한다:
   - 현재 장착된 페르소나가 있으면 해당 페르소나를 대상으로 제안한다
   - 없으면 보관소 목록을 제시하고 선택하게 한다
   - `~/.claude/personas/{slug}.md` 를 Read 도구로 읽는다

2. 새 자료를 수집한다:

```
📥 페르소나 자료 병합 모드

대상 페르소나: "{이름}"

추가할 자료를 입력해주세요:

① 텍스트 직접 입력
② URL (WebFetch로 내용 추출)
③ 파일 경로 (Read 도구로 읽기)

여러 형태를 동시에 입력해도 됩니다.
```

3. 자료를 수집 및 분류한다:
   - URL → WebFetch로 핵심 내용 추출
   - 파일 경로 → Read 도구로 파일 읽기
   - 텍스트 → 그대로 사용
   - 수집된 내용을 Layer 0~5 중 해당하는 곳으로 분류한다

4. 충돌 검사를 수행한다:
   - 새 자료가 기존 페르소나의 내용과 모순되는지 확인한다
   - **보강(새 디테일 추가)** → 해당 Layer에 append
   - **확인(기존 내용과 동일)** → 무시 (중복 기록하지 않음)
   - **충돌(기존과 모순)** → 유저에게 제시:

```
⚠️ 기존 내용과 충돌이 감지되었습니다:

기존: [기존 내용]
새 자료: [새 내용]

어떻게 처리할까요?
[1] 새 자료 우선 (기존 유지 + 새 자료를 추가 관점으로 병기)
[2] 기존 유지 (새 자료 무시)
[3] 직접 수정
```

5. 증분 패치 미리보기를 유저에게 제시한다:

```
📋 병합 미리보기:

[Layer 2: Expression Style]
+ 새 구어 표현 추가: "{예시}"

[Layer 3: Decision & Judgment]
+ 우선순위에 "{항목}" 추가

적용할까요? [Y] 확인  [N] 취소  [E] 수정
```

6. 유저가 `Y`를 선택하면:
   - `~/.claude/personas/{slug}.md` 를 Edit 도구로 수정한다
   - **기존 내용은 절대 덮어쓰지 않는다** — append only
   - v1 포맷(5-Layer가 아닌 기존 포맷)의 페르소나도 기존 섹션 구조를 유지하면서 추가한다

7. 현재 장착 중인 페르소나이면 CLAUDE.md도 갱신한다:
   - 수정된 보관소 파일을 다시 읽어서 `<!-- PERSONA:START -->` ~ `<!-- PERSONA:END -->` 블록을 교체한다

8. 완료 메시지:

```
✅ 자료가 페르소나에 병합되었습니다.

페르소나: {이름}
변경된 Layer: [Layer 번호 목록]
파일: ~/.claude/personas/{slug}.md
```

## Delete Flow — 페르소나 삭제

### 트리거

- "페르소나 삭제", "페르소나 지워줘"

### 절차

1. `~/.claude/personas/` 디렉토리를 Glob 도구로 스캔한다 (`~/.claude/personas/*.md`)
2. 파일이 없으면:

```
📭 삭제할 페르소나가 없습니다.
```

3. 파일이 있으면 목록을 제시한다:

```
🗑 페르소나 삭제

삭제할 페르소나를 선택해주세요:

| # | 이름 | 유형 | 생성일 | 파일 |
|---|------|------|--------|------|
| 1 | {name} | {type} | {created} | {slug}.md |
| ... |

번호를 입력해주세요. (취소: "취소")
```

4. 유저가 번호를 선택하면 확인을 요청한다:

```
⚠️ 정말 삭제하시겠습니까?

페르소나: "{이름}"
파일: ~/.claude/personas/{slug}.md

이 작업은 되돌릴 수 없습니다.
[Y] 삭제  [N] 취소
```

5. 유저가 `Y`를 선택하면:
   a. 현재 CLAUDE.md에 장착된 페르소나인지 확인한다
      - `~/.claude/CLAUDE.md` 의 `<!-- equipped: {slug} -->` 를 확인
      - 장착 중이면: `<!-- PERSONA:START -->` ~ `<!-- PERSONA:END -->` 블록을 먼저 제거한다
   b. `~/.claude/personas/{slug}.md` 파일을 Bash `rm` 명령으로 삭제한다
   c. 완료 메시지:

```
✅ 페르소나가 삭제되었습니다.

삭제된 페르소나: {이름}
{장착 중이었다면: "CLAUDE.md에서도 탈착되었습니다."}
```

## Checklist Before Delivery

### 생성 모드 (Creation Flow)
- [ ] G_data: 기질/도메인 특성이 구체적으로 수집되었는가
- [ ] H_data: 핵심 경험/이력이 수집되었는가 (생략 선택 포함)
- [ ] E_data: 주감정 + 보조감정 + 강도가 모두 설정되었는가
- [ ] R_data: 위기/시련 또는 실무 방법론이 수집되었는가
- [ ] [B] 실무형에서 WebSearch가 최소 1회 실행되었는가
- [ ] 페르소나 미리보기가 유저 확인을 거쳤는가
- [ ] `~/.claude/personas/{slug}.md` 에 보관소 파일이 생성되었는가
- [ ] frontmatter (name, type, created, version)가 올바르게 기록되었는가
- [ ] 5-Layer 구조(Layer 0~5 + Correction 기록)로 출력되었는가
- [ ] Layer 0이 동사형 행동 규칙으로 작성되었는가 (형용사 금지)
- [ ] Layer 2에 5개 시나리오별 대사 예시가 포함되었는가

### 장착 모드 (Equip Flow)
- [ ] 보관소에서 대상 페르소나 파일을 정상적으로 읽었는가
- [ ] CLAUDE.md의 기존 내용이 보존되었는가 (덮어쓰기 아님)
- [ ] `<!-- PERSONA:START -->` ~ `<!-- PERSONA:END -->` 블록이 올바르게 기록되었는가
- [ ] 기존 장착 블록이 있었다면 교체되었는가 (중복 방지)

- [ ] PERSONA 블록만 제거되고 나머지 CLAUDE.md 내용이 보존되었는가

### 교정 모드 (Correction Flow)
- [ ] 대상 페르소나가 확인되었는가 (장착 중 or 목록 선택)
- [ ] 교정 규칙이 동사형 행동 규칙으로 작성되었는가 (형용사 금지)
- [ ] 유저 확인을 거쳤는가
- [ ] 보관소 파일의 Correction 기록 섹션에 추가되었는가
- [ ] 장착 중이면 CLAUDE.md도 갱신되었는가
- [ ] 30개 초과 시 병합이 실행되었는가
- [ ] 5개 이상 시 Layer 0 승격 제안이 이루어졌는가

### 병합 모드 (Merge Flow)

- [ ] 유저 확인(미리보기)을 거쳤는가
- [ ] 장착 중이면 CLAUDE.md도 갱신되었는가

### 삭제 모드 (Delete Flow)
- [ ] 유저 확인을 거쳤는가
- [ ] 장착 중이었다면 CLAUDE.md에서 탈착되었는가
